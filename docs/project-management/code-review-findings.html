parseやcompileでは見えにくい、schema/generic tests、seed契約、過去のsingular test案の証跡境界を中心に見る。

F-01

率KPIの危険な再集計を固定するsingular test案が現candidateに残っていなかった

契約不一致
問題種別 契約不一致 構成図ノード dbt tests / KPI定義 発生ファイル 過去レビュー証跡: dbt/tests/singular/kpi_definitions_reaggregation_contract.sql案 / 現candidate: dbt/tests/singular/未配置
レビュー指摘

gross_margin_rate の分子分母再計算やカテゴリ率平均禁止を固定するsingular test案が過去レビューで出たが、このcandidate worktreeには実体ファイルがない。

影響

schema/generic testsがあっても、後段のSemantic/Evalが率KPIの誤集計を許す可能性は残る。

検知証跡

FB-077、FB-085の過去レビュー文脈と、現candidateで dbt/tests/singular/ が存在しないことの棚卸し。

再発防止

過去レビューのtest案を現物証跡として扱わず、実体化するまでfuture candidateとして表示する。

問題のある証跡扱い

HTML / current candidate evidence mismatch: docs/project-management/code-review-findings.html
<span class="tag-chip file">
  dbt/tests/singular/kpi_definitions_reaggregation_contract.sql
</span>

修正後の証跡扱い

HTML / evidence boundary: docs/project-management/code-review-findings.html
過去レビュー証跡:
  dbt/tests/singular/kpi_definitions_reaggregation_contract.sql案
現candidate:
  dbt/tests/singular/未配置

修正方針: 率KPIは「値が存在する」だけでなく、上位粒度で平均してはいけないという再集計ルールまでdbt contractに固定する。ただし実体ファイルがない間は、現物証跡ではなくfuture test candidateとして扱う。

F-09

修正方針: dbt staging/marts/testsは0%から低率へ上げる。tooltipでは実装証跡ありと未実施証跡を分ける。ただし、このcandidate worktreeでは dbt/tests/singular/ と dbt/target/ artifactは見つからないため、live dbt build/test、Snowflake実行、SQL結果正当性、answer-quality証跡は未実施として扱う。

F-10

率KPIの保存済み率をavg/sumする負例テストがまだ不足している

将来テスト候補
問題種別 reviewer-finding / 将来テスト候補 構成図ノード Semantic KPI Model / dbt tests 発生ファイル semantic/retail_kpi_semantic_model.yaml / tests/**
レビュー指摘

gross_margin_rate と avg_discount はmartで分子分母から再計算しているが、Semantic Modelや生成SQLが保存済み率を avg() / sum() しないことを壊す負例テストがまだない。

影響

構造化SQLやsemantic YAMLの変更で率KPIの誤集計が混入しても、現行のplanner/traceテストだけでは検出できない可能性がある。

検知証跡

L3 review finding。semantic/retail_kpi_semantic_model.yaml の率KPI定義と tests/test_planner_and_trace.py の現行assertを確認。

再発防止

次のtest-pattern更新で、保存済み率のavg/sumを禁止するnegative mutation testを追加候補として扱う。

不足が残る定義・テスト抜粋

YAML + pytest: semantic/retail_kpi_semantic_model.yaml / tests/test_planner_and_trace.py
- name: gross_margin_rate
  expr: gross_margin_rate
  default_aggregation: avg
- name: avg_discount
  expr: avg_discount
  default_aggregation: avg

assert plan.metric == "gross_margin_rate"
assert "MART_RETAIL_MONTHLY_KPI" in (plan.sql or "")

追加候補の負例テスト抜粋

pytest / future negative mutation candidate: tests/**
sql = (plan.sql or "").lower()

assert "avg(gross_margin_rate)" not in sql
assert "sum(gross_margin_rate)" not in sql
assert "avg(avg_discount)" not in sql
assert "sum(avg_discount)" not in sql

修正方針: blockingではなくfuture test candidateとして保持する。Semantic KPI Modelが保存済み率をそのまま平均・合計しないことを、負例mutationで壊れた時に失敗させる。

F-11

citation有無だけのevalでは、semantic根拠と文書根拠の不足を分けられなかった

評価境界
問題種別 評価境界 構成図ノード Golden Eval / Agent Router / Trace Store 発生ファイル data/golden_eval.json / retail_ai_ops/eval_runner.py / tests/test_planner_and_trace.py 文面所在 L2/L4統合候補 6d7fb0b
レビュー指摘

mixed queryや未対応KPIの評価で、単一の requires_citations と pass/fail だけでは、semantic YAML根拠はあるがdoc根拠は未接続、または現状routeはknown gapという状態を表せなかった。

影響

再発防止

semantic citation、doc citation、known gapのcurrent/desired routeを別フィールドで残し、local static evalが外部接続済み品質へ膨らまないようにする。

問題のある評価契約抜粋

JSON / Golden Eval fixture: data/golden_eval.json
"expected_answer_checks": {
  "answer_contains": ["SQL案は作成しました", "文書根拠取得はsafe-stopします"],
  "requires_sql": true,
  "requires_citations": true,
  "requires_human_review": true,
  "safe_stop_reason": "doc_source_missing"
}

修正後の評価契約・runner抜粋

JSON + Python / data/golden_eval.json and retail_ai_ops/eval_runner.py
"expected_answer_checks": {
  "requires_semantic_citation": true,
  "requires_doc_citation": false,
  "safe_stop_reason": "doc_source_missing"
}
"known_gap": {
  "current_route": "business_definition_question",
  "current_safe_stop_reason": "definition_lookup",
  "desired_route": "unsupported_question",
  "desired_safe_stop_reason": "semantic_model_missing"
}

"semantic_citations": _semantic_citations(answer.citations),
"doc_citations": _doc_citations(answer.citations),
"desired_safe_stop_reason": _known_gap_field(case, "desired_safe_stop_reason")

修正方針: L2/L4統合候補はreview済み候補として、Golden Evalとroutingの境界を細かく可視化した。semantic/retail_kpi_semantic_model.yaml の引用確認、doc citation未接続、known gapの現状route/望ましいrouteを分けるが、live Snowflake、Cortex Search、Pages公開、push済みの証跡としては扱わない。

並列セッション運用の改修実例

F-04

dispatch briefが整って見えても、新規worktreeが古いHEADを見るリスクがあった

ブランチリスク
問題種別 ブランチリスク 構成図ノード GitHub Repo / CI-CD 発生ファイル docs/project-management/parallel-session-implementation-plan.md
レビュー指摘

source docsやskillsが未commit、未trackingのままでは、別sessionが必要な設計を読めない。

影響

受理済みに見えるレーンが、実際には古い指示や古いskillで作業を進める。

検知証跡

FB-080。pre-dispatchレビューでdispatch base、branch、HEAD、push policy不足を検出。

再発防止

dispatch前にbase ref、worktree、branch、HEAD、対象commitを確認する。

問題のある記述抜粋

Markdown / 並列セッション親計画: docs/project-management/parallel-session-implementation-plan.md
ただし `accepted` / `reported` / `reviewed` / `integration_ready` はTerminalごとに
別々に判定する。1つのTerminalが報告しても、他のTerminalの受理や完了の証拠には
しない。

修正後記述抜粋

Markdown / 並列セッション親計画: docs/project-management/parallel-session-implementation-plan.md
dispatch前に正本docs/skillsが未コミットまたは未追跡の場合は、次のどちらかを選ぶ。

- dispatch base commit/branchを作る
- brief本文に必要な正本内容を省略せず含める

別worktreeがHEADやbase_refから作られる場合、未コミット正本は読めないため、
この確認がないまま `accepted` へ進めない。

修正方針: 状態語の分離だけでは足りないため、別worktreeが読む正本commitを明示し、未commit正本はbrief本文に完全同梱させる。

F-05

レビュワーが integration_ready を結論として使いかけた

状態語混同
問題種別 状態語混同 構成図ノード GitHub Repo / LLMOps運用 発生ファイル skills/ai-architecture-learning/child-skills/code-review-ai-mistake-patterns/SKILL.md 文面所在 FB-081 / レビュワー結論
レビュー指摘

レビュワーlaneが最終統合判断に近い語を使うと、main sessionの独立確認を飛ばしやすい。

影響

reviewedとintegration_readyが混ざり、blocking findingやbranch鮮度確認を抜かす。

検知証跡

FB-081。レビュワーはverdictやrecommendationだけを返すルールに修正。

再発防止

統合可能判断はmain sessionだけが行い、reviewerは reviewer_verdict に留める。

問題を示すレビュー記録

Markdown table / Feedback Ledger reviewer finding: .agent-feedback/FEEDBACK_LEDGER.md#FB-081
FB-081 | The reviewer lane used `integration_ready` as a reviewer conclusion even though that state belongs only to the orchestrator after independent review.

修正後ルール抜粋

Skill Markdown / reviewer rule: skills/ai-architecture-learning/child-skills/code-review-ai-mistake-patterns/SKILL.md
Reviewer sessions must not declare `integration_ready`. That state belongs to
the orchestrating main session after independent artifact review, checks,
branch/worktree inspection, and blocking-finding triage.

修正方針: reviewerは結論をrecommendationに留め、main sessionだけが成果物確認、checks、branch状態を見て統合可否を決める。

ページ分離ルール

  • 総合ビューはレビュー観点、状態語、証跡レベル、反映先を扱う
  • ソースコード改修実例ビューは実レビュー由来の改修実例だけを扱う
  • 抽象カテゴリだけの項目はソースコード改修実例ビューに入れない
  • 新しい実findingが出たら、このページへ追加するか、別ページ化を検討する

  • 修正済みでも、何を防ぐべきだったかを残す
