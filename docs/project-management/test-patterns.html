Test Pattern Catalog

テストパターン一覧

Retail AI Ops Copilotで使うべき検証パターンを、実行結果ではなく「何を証明するか」「何を証明しないか」の単位で整理する。 static、単体/ユニット、contract、local data、compile、結合、E2Eを混ぜず、後続のSemantic Model、Golden Eval、Streamlit、Cortex readinessへ渡せる形にする。

レビュー面入口へ戻る 構成図を見る ソースコード改修実例を見る AIミスパターンを見る 設計ソース: test-pattern-reuse skill instruction。Pagesブランチではskill本文へリンクしない。

位置づけ

このHTMLはGitHub Pagesで人間が読むためのカタログであり、単独ではテスト成功証跡ではない。 実施済み、未実施、環境未整備、live未証明を明示し、main sessionの統合判断を補助する。

HTML = test-pattern map
commands/artifacts = proof surface
main session = promotion decision owner

証跡レベル

同じ「テスト」でも、証明できる範囲が違う。readiness報告ではレベル名と限界を一緒に残す。

証跡: static-lint ファイル、HTML、docs、ledger、グラフ構造を読む。実サービスや実データの成功は証明しない。
証跡: fixture-unit / 単体・ユニットテスト local fixtureやmock入力に対する決定的テスト。関数、変換、通知生成などの小さい単位を検証し、外部配送やlive APIの成功とは分ける。
証跡: contract-test seed、model、API、workflow state、eval caseなどの契約を検査する。契約外の品質は別テストが必要。
証跡: data-bearing-local 現在のseed、fixture、出力値を使う。local dataの妥当性とlive warehouseの妥当性は同一ではない。
証跡: compile-only dbtやツールが構文解析、依存解決、生成をできることを示す。live実行成功にはしない。
証跡: live-integration / 結合テスト 対象service、role、schema、warehouse、成功/失敗ログがそろう実結合証跡。dummy envや単体テストとは別扱い。
証跡: user-facing-e2e / E2Eテスト 人間が見る画面、回答品質、承認フローまで検証する。内部関数、単体テスト、compileだけでは代替できない。
証跡: workflow contract 状態語、報告責務、反映手順などの運用契約を検査する。実配送や成果物レビューの代替ではない。
証跡: eval pattern Golden Evalや回答品質回帰の検査設計。実行済みE2E結果とは分けて扱う。

状態語と証跡境界

並列セッションとremote issue運用では、配送、受理、報告、レビュー、統合判断を同じ言葉で扱わない。

posted / delivered

タスクやpane送信が存在する段階。担当sessionが読んで作業を受けた証拠ではない。

accepted / reported

担当sessionが受理または報告した段階。成果物の正しさや統合可否はまだ別確認。

reviewed / integration_ready

監督側レビューと統合判断の語。lane単独のHTML実装では宣言しない。

パターン一覧

各パターンは「使うべきか」「証明すること」「証明しないこと」を同じ粒度で見る。

カテゴリ: dbt readiness 証跡: compile-only

dbt parse / compile --empty

現在のプロジェクト状況

credentials-freeのdummy envでdbt projectを構文解析、空compileする入口がある。RAIOPS-4系の前段品質gateとして使う。

使うべきか

使う。dbt編集の最初の安全確認。ただしlive Snowflake proofの代替にしない。

コマンドまたは証跡artifact

dbt parse --project-dir dbt --profiles-dir /tmp/retail-ai-ops-dbt-profiles
dbt compile --no-populate-cache --no-introspect --empty

何を証明するか

YAML、Jinja、macro、selector、test定義がparse/compile可能で、reviewableなdbt artifactへ進めること。

何を証明しないか

live Snowflake接続、warehouse上の行数、role権限、dbt test成功、Cortex/Semanticの回答品質。

再利用先

project-family、generic client-delivery skill、Backlog受入基準、docs template。

関連ノードまたはレーン

dbt staging/intermediate/marts、RAIOPS-4 dbt品質gate、Semantic Model readiness。

次アクションまたはブロッカー

live資格情報を使う別laneでdbt seed/build/testを実行し、target、role、schema、失敗test名を残す。

カテゴリ: KPI contract 証跡: contract-test 証跡: data-bearing-local

dbt generic / singular tests

現在のプロジェクト状況

seed契約、model schema、generic tests、KPI seed行、unsupported gap、reaggregation hazard用のsingular testがある。

使うべきか

使う。Semantic ModelとGolden Evalへ渡すKPI定義、grain、率系/再集計リスクの最低契約を守る。

コマンドまたは証跡artifact

dbt/seeds/schema.yml
dbt/tests/singular/*.sql
dbt test --select tag:llm_contract+

何を証明するか

必須KPI行、not_null/unique、accepted values、unsupported/gap表現、再集計注意の契約が壊れていないこと。

何を証明しないか

LLMが正しく説明すること、live warehouseで全testがpassすること、rate KPIを月別以外へ安全に再集計できること。

再利用先

project-local、project-family、lint・CI、Backlog受入基準、generic client-delivery skill。

関連ノードまたはレーン

kpi_definitions seed、mart_retail_monthly_kpi、Semantic input、Eval input。

次アクションまたはブロッカー

live dbt testを別証跡で追加し、率系KPIの再集計はSemantic/Golden Eval側のケースへ接続する。

カテゴリ: notification workflow 証跡: fixture-unit / 単体・ユニット

Backlog / Slack private_simulation Node tests

現在のプロジェクト状況

repo familyではprivate_simulationの通知テストパターンを扱う。このbranch単体ではmain統合前のlane artifactが未搭載の場合がある。

使うべきか

使う。遠隔報告やBacklog/Slack文面の決定的テストとして有効。ただし実配送の証跡語と分ける。

コマンドまたは証跡artifact

node --test tests/backlog_slack_notification.test.mjs
--mode private_simulation
sent_external=false

何を証明するか

fixture入力から本文、重複抑止、宛先メタデータ、外部送信しない安全経路が期待通りになること。

何を証明しないか

実Slack投稿、実Backlog API delivery、相手paneのaccepted、GitHub Issue上のreported確認。

再利用先

project-family、generic client-delivery skill、Backlog受入基準、docs template。

関連ノードまたはレーン

remote issue reporting、Backlog/Slack通知lane、parallel session operations。

次アクションまたはブロッカー

main統合後に現物ファイル名でコマンドを固定し、実配送確認とは別のlive-integration項目を作る。

カテゴリ: diagram quality 証跡: static-lint

diagram quality lint

現在のプロジェクト状況

connector geometry、text layout、diagram qualityをまとめて見るNode wrapperがある。構成図HTMLの変更時に必須gateとして扱う。

使うべきか

使う。GitHub Pagesの構成図品質を安定させるためのcheapで決定的な静的gate。

コマンドまたは証跡artifact

node tools/check_diagram_quality.mjs
tools/check_diagram_connectors.mjs
tools/check_diagram_text_layout.mjs

何を証明するか

HTML/SVGの接続線、矢印、ラベル、テキストoverflowなどのページ品質ルールが壊れていないこと。

何を証明しないか

ビジネスロジック、dbt契約、source imageとの芸術的忠実度、Snowflake/Cortex実行成功。

再利用先

lint・CI、project-family、generic client-delivery skill、docs template。

関連ノードまたはレーン

architecture HTML、diagram review lane、readiness overlay。

次アクションまたはブロッカー

カテゴリ: feedback workflow 証跡: static-lint 証跡: workflow contract

feedback reflection lint

現在のプロジェクト状況

使う。reviewer findingや改善手順を反映する変更では必須。今回のHTML laneでは編集禁止のため提案に留める。

関連ノードまたはレーン

次アクションまたはブロッカー

カテゴリ: parallel session workflow 証跡: static-lint 証跡: workflow contract

parallel session guardrail lint

現在のプロジェクト状況

カテゴリ: Python runtime 証跡: fixture-unit / 単体・ユニット 証跡: data-bearing-local

Python pytest planner / trace / Obsidian sync

現在のプロジェクト状況

tests/test_planner_and_trace.pyとtests/test_sync_obsidian_docs.pyがある。pytest-capable envがない場合はenvironment-not-readyとして扱う。

使うべきか

再利用先

関連ノードまたはレーン

次アクションまたはブロッカー

このプロジェクトで使う順序

最初からすべてをlive E2Eにしない。安い検知で契約を固め、liveとuser-facingは別証跡として積む。

1. static / compile

HTML、docs、diagram、dbt parse/compileで構造破損を先に止める。

2. contract / fixture

seed、mart、notification、workflow stateの契約をlocal fixtureで固定する。

3. live integration

Snowflake、Backlog、Slack、Cortexなどはtargetと実ログを添えて別証跡にする。

4. user-facing eval

Golden Eval、Streamlit、Cortex回答の品質は、ユーザーが見る経路として検証する。
