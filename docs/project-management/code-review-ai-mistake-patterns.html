状態語は段階ごとに止める

pane配送、受理、報告、レビュー、統合判断は別の証跡を必要とする。

posted タスク文がIssueやdispatch surfaceに存在する。相手が読んだ証拠ではない。
delivered bridgeやpane送信が完了した。Codexが受け入れた証拠ではない。
accepted 受信transcript、worktree、明示返信などで作業受理が確認できる。
reported 担当sessionが完了または状況を返した。内容の正しさは未レビュー。
reviewed 監督側がファイル、diff、出力、検証ログを独立に見た状態。
integration_ready blocking findingがなく、branch鮮度とchecksを確認した後にmain sessionだけが判断する。

証跡レベルを混ぜない

static/mock/dummy/localは有用な証跡だが、live data-bearing proofとは扱いを分ける。

static

構文と生成物

`dbt parse`、compiled SQL、manifest、DOM textなど。構造が読める証跡。

mock / dummy

代替入力

dummy credentials、fixture、`--empty`、モック。live経路の代替成功にはしない。

local review

人間の突き合わせ

CSV行、生成SQL、画面文言、diffを読む。過大主張と契約ズレを止める。

live data-bearing

実データ実行

target、role、warehouse、database、schema、成功/失敗test名を残す。

AIミスカテゴリ別の見方

findingはカテゴリ、症状、証跡、質問、反映先まで一緒に返す。

証跡の過大主張

静的成功をlive成功に見せる

典型症状

`parse`、`compile --empty`、mockの成功を、実データやwarehouse品質の成功として報告する。

検知証跡

コマンド、env、target、compiled SQL、実行ログ、live target情報の有無。

レビュー質問

これはstatic、dummy、local、liveのどれか。liveならrole/schema/test名は残っているか。

反映先

dbt-readiness-gates、Backlog ticketing、deterministic check。

契約不一致

テスト期待値が実データとずれる

典型症状

singular testは生成できるが、seed row、fixture、mart grainの現物と一致しない。

検知証跡

CSV行、schema.yml、compiled test SQL、manifest、Golden Eval case。

レビュー質問

期待語彙は現在のartifactに存在するか。重要な注意書きはテストで守られているか。

反映先

domain child skill、row-level lint、CI gate、source contract doc。

成果物不具合

成果物そのものが壊れている

典型症状

リンク切れ、HTML構文崩れ、selector漏れ、意図したタグや列がmanifestに出ない。

検知証跡

ファイル存在、リンク先、HTML parser、`dbt ls`、manifest、ブラウザ表示。

レビュー質問

報告だけでなく、実ファイル、生成物、レンダリング結果を見たか。

反映先

deterministic test/lint、artifact review handoff、対象domain skill。

文書陳腐化

設計書と実装状態が古いまま残る

典型症状

pre-change件数とpost-change件数が混在し、どの証跡時点の数字か分からない。

検知証跡

Backlog本文、設計doc、README、Obsidian mirror、最新コマンド結果。

レビュー質問

baseline、実装後、未実施、次のlive証跡が明確に分かれているか。

反映先

Backlog ticketing、doc sync gate、project rule、feedback ledger。

ブランチ・統合リスク

branch状態を見ずに統合できると言う

典型症状

`ahead` と `behind`、shared file編集、許可外ファイル変更を見落とす。

検知証跡

`git status --short --branch`、changed-file list、base_ref、diff scope。

レビュー質問

このlaneは何を変更してよいか。main session専有ファイルに触れていないか。

反映先

parallel-session review orchestration、task brief contract、worktree integration。

レビュー工程の穴

今回見つけた穴を次回も見逃す

典型症状

findingだけ返し、どのskill、lint、Backlog ruleへ反映するか残さない。

検知証跡

review report、feedback ledger、child skill、CI/lint、acceptance criteria。

レビュー質問

これは一回限りか、再利用できるミスパターンか。次にどこで止めるか。

反映先

code-review-ai-mistake-patterns、domain child skill、deterministic lint。

レビュー時に必ず聞く質問

報告を読んだだけで終えず、artifact、証跡レベル、反映先を分けて確認する。

何が変わったか

changed files、diff、生成物、画面を見たか。報告だけを信用していないか。

証跡はどのレベルか

static、mock、dummy、local、live data-bearing proofを分けて書けるか。

契約と現物は合うか

seed row、fixture、schema、Semantic/Eval名、mart grainが一致するか。

テストは壊れるべき時に壊れるか

現在の正しいartifactでは通り、意図した欠陥では落ちるか。

branchは統合可能か

clean/current/allowed-file内か。`ahead behind` を無視していないか。

次回どこで防ぐか

skill、Backlog rule、lint、CI、ledgerだけ、のどれに返すか。

反映先を選ぶ

findingは修正提案だけで終えず、再発防止の置き場所まで分類する。

反映先 使う場面 注意点
code-review-ai-mistake-patternsAI/並列agentに共通するレビュー観点を増やす。domain固有なら子スキルへ逃がす。
domain child skilldbt、Backlog、diagramなど具体領域の受入基準を強める。親skillだけに汎用文として埋めない。
