# 06_セキュリティ・コスト境界

## 基本方針

セキュリティとコストは軽視しない。ただし本プロジェクトでは、企業レベルのIAMやFinOps自動化を作るのではなく、AIアーキテクトとして破ってはいけない境界を実装可能な形で示す。

## Security boundary

初期ロール:

- `store_manager`
- `regional_manager`
- `analyst`
- `admin`

MVPで確認すること:

- roleに応じて見える店舗/地域が変わる
- LLM-safe fields 以外を回答contextへ入れない
- unsupported / sensitive question を拒否できる
- action request は human approval に送る
- traceには必要十分な監査情報を残し、秘密情報は残さない

深掘りしないこと:

- SSO/OIDC本実装
- enterprise DLP
- SIEM連携
- key rotation
- SOC2 evidence collection
- 複雑なrow-level securityの完全再現

## Cost boundary

学習環境での暴走を防ぐ。

MVPで扱うこと:

- XS warehouse / auto suspend 前提
- query timeout
- max result rows
- answer context の最大行数
- model/message count logging
- Cortex Search service の停止手順
- query tag
- resource monitor の設計メモ

深掘りしないこと:

- 完全なSnowflake spend automation
- 部門別配賦
- 契約単価ベースの精密試算
- production FinOps dashboard

## Cost-relevant trace fields

- warehouse_name
- query_tag
- row_count
- result_cache_hit
- tool_name
- model_name
- prompt_version
- message_count
- latency_ms
- search_service_name

## 重要な考え方

コスト制限は本番設計を歪めるためのものではない。

学習環境では小さいデータ、小さいwarehouse、少ないE2Eで抑える。一方で、Agent state、trace、eval、conversation reuse、human review は削らない。ここを削ると、実業務でのLLMOps学習価値が落ちる。
