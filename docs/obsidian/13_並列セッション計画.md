# 13_並列セッション計画

Repo mirror: [Parallel Session Implementation Plan](../project-management/parallel-session-implementation-plan.md)

## 結論

並列sessionで実装する前に、設計、Backlog ticket粒度、受入基準、編集範囲、
検証コマンド、共有ファイルの統合責任を先に固める。

現時点では、すぐに複数sessionで実装を開始しない。まず `RAIOPS-4` の
dbt tests / readiness gate設計を固め、その後に `dbt品質gate`、
`Semantic契約`、`Golden Eval拡張` を最初の並列候補として具体化する。

## 並列化の原則

- 1 sessionは原則1 laneまたは1 Backlog ticketを担当する
- shared docs、feedback ledger、project skill、構成図HTML、Obsidian mirrorは
  main sessionが統合する
- 実装sessionは、受入基準と検証コマンドが明記されたあとに開始する
- Snowflake/Cortex/Snowpark live挙動ではsilent fallbackしない

## 初期候補

| lane | 対応ticket | 目的 |
| --- | --- | --- |
| dbt品質gate | `RAIOPS-4` | dbt testsをSemantic/Eval/readinessの前提にする |
| Semantic契約 | `RAIOPS-5` | Semantic Model YAMLとKPI定義seedをdbt mart契約に合わせる |
| Golden Eval拡張 | `RAIOPS-6` | route/metricだけでなく回答品質とunsupported判断を見る |
| Cortex live trace | `RAIOPS-7` | live pathのtraceを残す |
| Agent / HITL | `RAIOPS-8`, `RAIOPS-9` | routing、safe-stop、承認queueを検証する |

## 次の一手

`RAIOPS-4` の設計を先に固める。

その後、並列session用のbriefを作り、編集してよいファイルと触らないファイルを
明示してから実装に入る。
