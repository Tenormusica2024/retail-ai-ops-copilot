# 01_学習目的・スコープ

## 目的

AIアーキテクト / LLMOps エンジニアとして、実装を伴う設計会話に入れる状態を作る。

このプロジェクトで作るものは、単なるチャットボットではない。構造化された小売KPIデータを、LLM / AIエージェントが安全に参照し、回答の品質を評価し、改善を継続できる小さな実装である。

## 学習の主戦場

| 領域 | 本プロジェクトでの扱い |
|---|---|
| LLMOps | golden eval、trace、version registry、error taxonomy、human review を中核にする |
| AIアーキテクチャ | Snowflake-native な semantic KPI access と agent orchestration を設計する |
| AIエージェント強化 | state、tool routing、fallback、approval、監査ログを実装対象にする |
| データ基盤 | curated mart を前提に、dbt/semantic model/契約だけ扱う |
| セキュリティ | role boundary、LLM-safe fields、denied-path test まで扱う |
| コスト | 学習環境の暴走防止と cost-relevant metadata まで扱う |

## 成功条件

最初の成功条件は以下。

> semantic model、prompt、agent routing を変更したとき、KPI回答が良くなったか悪くなったかを検知できる。

そのため、MVPの見た目より先に、以下を作る。

- 代表的なKPI質問の golden eval
- route / plan / SQL / result / answer / error を残す trace
- prompt、semantic model、dataset、Cortex設定の version registry
- 失敗を分類する error taxonomy
- 低信頼・失敗ケースの human review loop

## 初期対象にしない成功条件

以下は初期成功条件にしない。

- 生データを綺麗にする前処理パイプラインの完成
- 需要予測モデルの精度競争
- 企業レベルのIAM/SSO/DLP実装
- Snowflakeコスト最適化の完全自動化
- 大規模RAGや全文書検索基盤
- 自律的な本番書き込みアクション

## 役割境界の仮説

Snowflake/dbt/AI/生成AI/AI Agent の公開情報から、構造化データとAIエージェントを接続する設計には学習価値がある。実務的な学習題材としては、DE/DSが整えたデータを、LLMやAIエージェントがどう安全に使うかに寄せる。

このため本プロジェクトは、以下の前提を置く。

- upstream cleansing は DE/DS 領域として扱う
- curated KPI marts と data contracts を入力にする
- AI側では semantic model、tool routing、評価、監査、改善ループを設計する
- セキュリティとコストは、AI側が破ってはいけない境界として扱う
